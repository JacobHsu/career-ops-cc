# 模式：104 推薦信（自我介紹信）

用於產生可直接複製貼上至 104 應徵頁面「自我介紹」欄位的繁體中文信件。

**不要替使用者送出。只產生可複製的文字，由使用者確認後自行貼入。**

## Step 0 — 前置條件檢查

在讀取任何資料前先驗證：

1. **繁中 CV**：檢查 `cv.zh.md` 是否存在
   - 存在 → 使用 `cv.zh.md`
   - 不存在 → 提示：「請先建立 cv.zh.md（繁中履歷）再產生推薦信」，並停止

2. **JD 資料**（當使用者提供 104 URL 時）：
   - 從 URL 末段提取 job_id（例：`https://www.104.com.tw/job/8xxxxx` → job_id = `8xxxxx`）
   - 檢查 `jds/{job_id}.md` 是否存在
     - 存在 → 自動讀取，標記來源為 `jds/{job_id}.md`
     - 不存在 → 提示：「尚未抓取此職缺 JD，請先執行 fetch-jd 或直接貼上 JD 文字」

## 輸入

使用者提供下列任一項：
- 104 職缺網址（優先，自動讀取對應 `jds/` 檔案）
- 貼上的 JD 文字
- 公司名稱 + 職稱（最低限度）

## 流程

1. **Step 0**：執行前置條件檢查（見上方）
2. 從 JD 擷取：公司名、職稱、主要職責、必要技能、加分條件
3. 讀取資料來源（見下方）
4. 如果 `reports/` 已有該職缺的評估報告，一併讀取（補充 fit 分析）
5. 產生自我介紹信（見下方結構）
6. 標出任何需要使用者填入的空白處

## 資料來源

| 來源 | 優先順序 | 說明 |
|------|----------|------|
| `cv.zh.md` | 必讀（主要） | 繁中 CV，所有經歷與數字的唯一來源 |
| `cv.md` | fallback | 僅在 `cv.zh.md` 不存在時使用 |
| `jds/{job_id}.md` | 必讀（若有） | fetch-jd 抓取的完整 JD 結構 |
| `config/profile.yml` | 必讀 | 個人定位、薪資、narrative |
| `modes/_profile.md` | 必讀 | 使用者個人化設定 |
| `article-digest.md` | 若存在 | 補充 proof points |
| `reports/{num}-*.md` | 若存在 | 補充評估分數與 fit 分析 |
| `examples/coverletter/coverletter-examples.md` | 若存在 | 業界寫作模式 + 使用者成功範例，作為文風與結構參考 |

**規則**：所有經歷、數字、技術只能來自 `cv.zh.md`（或使用者明確提供的資訊），不可憑空填入。

## 信件結構

四段，總長 250–350 字，繁體中文：

**第一段（定位，2–3 句）**
- 說明自己是誰、幾年資歷、核心專長
- 點出與這個職缺最直接相關的一個身份

**第二段（匹配，3–4 句）**
- 對照 JD 的主要職責，舉出最相關的一段工作經歷或專案
- 用具體數字或成果（從 `cv.zh.md` 取，不要編造）

**第三段（動機，2–3 句）**
- 說明為什麼對這家公司或這個職位有興趣
- 如果 JD 有明確產品或技術方向，具體呼應

**第四段（收尾，1–2 句）**
- 期待有機會進一步說明
- 不要用「感謝您百忙中撥冗閱讀」這類空泛套語

## 語氣守則

- 直接、專業，不誇大
- 技術詞彙保留英文（React、CI/CD、LLM 等）
- 不用「熱愛學習」「積極主動」「抗壓性高」
- 不用第一人稱「我」開頭每一句
- 避免翻譯腔

## 輸出格式

```
## {Company} {Role} — 推薦信

{信件正文，直接可複製}

---
字數：{N} 字
來源：cv.zh.md + jds/{job_id}.md（+ report {num}，如有）
需確認：{如有空白或不確定處，列於此}
```

## 注意事項

- 所有經歷、數字、技術只能來自 `cv.zh.md` 或使用者明確提供的資訊，不可憑空填入
- 如果 JD 資訊不足（例如只有職稱），先問使用者要補充哪些重點，再產生
- 如果使用者對輸出不滿意，詢問哪個段落需要調整，不要整封重寫
